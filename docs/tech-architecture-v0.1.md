## 1. 结论

当前项目的 MVP 技术路线建议直接定为：

- **引擎 / 框架：PhaserJS + TypeScript**
- **表现形式：2D isometric / faux-2.5D（伪 2.5D 斜视角）**
- **原型范围：客户端单机战斗原型 / 伪联机验证，不上正式服务端**
- **原型重点：战斗沙盒 + 任务 + 撤离 + 最简养成**
- **内容组织：从第一版开始做轻量数据驱动**

## 2. 关于 FFTA 的视觉分类

### 2.1 结论
**FFTA 本质上是 2D isometric / faux-2.5D，不是真 3D。**

更准确地说：
- 地图是**等距/斜视角的 2D 表现**
- 角色是 **2D sprites**
- 通过高度、透视和地形层级制造出“像 2.5D”的视觉感觉
- 但它不是需要真 3D mesh、真 3D 摄像机的那种系统

### 2.2 为什么这么判断
公开资料里有两类信息能支撑这个结论：
- 评测和资源站直接把 FFTA 描述为 **isometric grid maps + 2D sprites**
- 逆向资料里可以看到 FFTA 地图数据本身包含 tile arrangement 和 **height-map data**，说明它的“立体感”核心来自 **2D 地图 + 高度数据**，而不是真 3D 场景

### 2.3 对你项目的含义
这对你是好消息。

因为你不需要追求：
- 真 3D 地形
- 真 3D 摄像机
- 复杂 3D 碰撞
- 重度模型管线

你真正需要的是：
- **格子逻辑**
- **高度数据**
- **等距/斜视角渲染**
- **遮挡排序**
- **Move / Jump / 击退 / 坠落规则**

也就是说：

> **你的目标不是做 3D FFTA，而是做像 FFTA 那样“看起来有立体层次的 2D 战棋”。**

## 3. 为什么 PhaserJS 适合这个项目

## 3.1 最重要的理由：你熟 JS
MVP 阶段，开发速度比“理论最强引擎”更重要。

你熟 JS，就意味着：
- 规则迭代更快
- 原型成本更低
- UI、调试工具、配置系统更好做
- 后续做数据驱动和编辑工具更顺手

## 3.2 Phaser 足够支撑当前目标
你当前要验证的是：
- Move / Jump
- 朝向
- 连携
- 击退
- 任务揭示
- 撤离
- 底盘 + 模块

这些都是**战斗规则问题**，不是重引擎问题。

Phaser 做下面这些是够用的：
- tile/grid 逻辑
- sprite 渲染
- UI 交互
- 动画演出
- 配置驱动
- 客户端单机场景组织

## 3.3 当前阶段不该为未来联机过度设计
你已经明确：
- MVP 先不做服务端
- 联机放后面
- 先做单机战斗原型或伪联机

所以现在不需要因为未来可能有正式多人，就先把技术栈换成更重的方案。

## 4. PhaserJS 的边界

## 4.1 它能做什么
PhaserJS 很适合：
- 2D 等距格子地图
- sprite 排序
- 点击选格
- 路径高亮
- 回合制状态机
- 动画和特效
- 战斗 UI
- 浏览器内快速迭代

## 4.2 它不替你解决什么
Phaser 不会自动替你解决：
- 等距坐标系统
- 高低差遮挡
- 真正的战棋规则层
- 正式多人同步
- 权威服务器
- 防作弊

但这些里，当前阶段真正需要你解决的，其实只有：
- 坐标系统
- 高低差表现
- 战斗规则状态机

这仍然在可控范围内。

## 4.3 什么时候 Phaser 会开始吃力
后面如果你要做：
- 重度多人实时同步
- 超复杂特效
- 真 3D 地形
- 大量角色同屏
- 端游级长线内容

那时再重新评估引擎都来得及。

现在不需要为那个未来过早付出开发效率。

## 5. MVP 视觉技术路线

## 5.1 推荐路线
**2D isometric / faux-2.5D**

也就是：
- 战斗逻辑是纯网格战棋
- 地图格子有 `x / y / height`
- 渲染层按等距坐标把格子画成斜视角
- 高度通过垂直偏移、台阶贴图、阴影和排序表现

## 5.2 不建议第一版做真 3D
不要一开始就做：
- 3D mesh 地形
- 旋转摄像机
- 真实 3D 碰撞
- 模型驱动角色

因为这些不会增加你 MVP 最需要验证的东西，反而会拖慢节奏。

## 5.3 推荐表现关键词
如果后续要统一描述，建议用：
- **FFTA-like isometric 2D tactics**
- **faux-2.5D isometric grid tactics**
- **2D sprites + height-driven isometric terrain**

这样比直接说“做 2.5D”更准确。

## 6. 场景与地图实现建议

## 6.1 地图逻辑结构
每个格子至少应包含：
- `x`
- `y`
- `height`
- `terrainType`
- `walkable`
- `cover/vision flags`（可后加）
- `event marker`（补给点、机关、危险区等）

## 6.2 渲染结构
建议地图渲染拆成：
- 地板层
- 高度层 / 台阶边缘层
- 交互物层
- 单位层
- 特效层
- UI 高亮层

## 6.3 单位排序
单位显示顺序建议至少由这些值组合决定：
- `tileY`
- `tileX`
- `height`
- 单位脚底锚点

这样才能避免高低差下的错误遮挡。

## 6.4 高低差表现
首版不需要超精美，但必须可读：
- 用明显的高台边缘
- 用高度阴影或边缘线
- 用格子选中高亮告诉玩家“这里能不能到”
- Move 和 Jump 结果要直接可视化

## 7. 原型开发顺序

## 7.1 第一阶段：战斗沙盒
只做：
- 单张测试地图
- 2 队 3 人
- Move / Jump
- 朝向
- 普通攻击
- 击退
- 轻量连携
- 基础行动序

目标：
- 先验证战棋手感成立不成立

## 7.2 第二阶段：任务 + 撤离
加入：
- 30 秒主任务揭示
- 1~2 个任务模板
- 撤离点
- 强制收口
- 局内收益

目标：
- 验证“战斗 + 任务 + 撤离”闭环

## 7.3 第三阶段：最简养成
加入：
- 3~5 个底盘
- 少量模块
- 编队界面
- 局后解锁

目标：
- 验证构筑层是否成立

## 7.4 第四阶段：伪联机 / 双队验证
再考虑：
- 本地双队模拟
- 双方信息遮罩
- 伪联机流程
- 轻量房间原型

目标：
- 验证多人关系模型，而不是先做正式服务端

## 8. 数据驱动方案

## 8.1 什么叫数据驱动
数据驱动的意思是：

**底盘、模块、任务、地图事件这些规则尽量写在配置里，而不是硬编码在战斗代码里。**

比如不写成：
- `if class === 'vanguard' then move = 4`

而是写成：
- 一个 `vanguard.json` / `vanguard.ts config` 来定义 Move、Jump、槽位、特性

## 8.2 为什么这个项目必须做数据驱动
因为你这个项目后面一定会频繁改：
- 底盘数值
- 模块效果
- 连携范围
- 任务目标
- 撤离条件
- 补给点刷新规则

如果这些全写死在代码里，后面会很痛苦。

## 8.3 MVP 必须数据驱动的内容
第一版至少把下面 4 类做成配置驱动：

1. **底盘**
2. **模块**
3. **任务模板**
4. **地图事件 / 补给点刷新规则**

## 8.4 配置形式建议
因为你熟 JS/TS，MVP 推荐这样做：
- 用 **TypeScript 配置对象** 或 **JSON**
- 不必一开始引入复杂外部表系统
- 优先保证：
  - 改参数不用动核心逻辑
  - 新增一个底盘/模块/任务模板成本低

### 推荐目录思路
- `src/data/chassis/`
- `src/data/modules/`
- `src/data/missions/`
- `src/data/map-events/`
- `src/data/units/`（后续可选）

## 8.5 为什么不建议第一版上复杂编辑器
因为你现在最需要的是：
- 快速改
- 快速测
- 快速否定错误方向

不是先花很多时间做工具链。

## 9. 客户端优先架构建议

## 9.1 当前阶段结论
MVP 先做：
- **纯客户端本地战斗状态机**
- **本地任务逻辑**
- **本地撤离与结算**
- **本地存档/配置**

## 9.2 为什么这样更合理
你当前最需要验证的是：
- 战斗系统成立吗
- 任务揭示成立吗
- 撤离张力成立吗
- 构筑层成立吗

这些都不依赖服务端。

## 9.3 后续联机怎么接
未来如果要上正式多人，最好改成：
- 客户端只做展示和输入
- 服务端做权威战斗状态

但这一步等玩法闭环跑通后再做。

这个项目当前最合理的技术路线不是：
- 上重引擎

这是最符合你当前目标、经验和项目阶段的路线。
