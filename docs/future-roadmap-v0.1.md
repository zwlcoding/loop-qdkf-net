## 1. 当前共识

当前 MVP spec 已完成，已经验证：

- 战斗底座成立
- 任务揭示成立
- 撤离收口成立
- 底盘 + 模块 + 最小成长成立
- 前端原型可运行、可测试、可 build

所以后面不该继续盲目加底层规则，
而应优先验证这个项目最独特的部分：

> **多方关系下的战术博弈：玩家队伍、bot 队伍、野怪、Boss、任务目标、撤离压力如何共同成立。**

---

## 2. 路线总原则

### 2.1 先验证玩法不确定性，再加重系统
优先级应是：

1. 先验证多人关系模型是否好玩
2. 再验证 PvE 压力是否成立
3. 再扩内容生产
4. 最后再上重系统（真联机、LLM Boss、复杂生成、视觉升级）

### 2.2 先做规则层，再做智能增强层
当前所有非玩家行为先走：

- 规则评分层
- 数据驱动 profile
- 可测试、可重放、可调参

不要现在就把核心战斗体验绑到：

- 真实联网
- 大语言模型实时决策
- 复杂程序化地牢生成

### 2.3 先做高频自测能力
后续优先级必须服务一件事：

> **让你一个人就能高频验证合作 / 对抗 / 反转。**

---

## 3. 当前要做：下一阶段主线

## 3.1 第一优先：统一非玩家行为系统
目标：
建立一套统一 AI foundation，服务三类对象：

1. bot 队伍
2. 野怪
3. Boss

### 为什么排第一
- 这是伪联机验证的基础
- 这是 PvE 压力的基础
- 这是后续 LLM Boss 的护栏基础
- 这是未来低活跃期 bot 填充、练习模式、守卫单位的复用基础

### 当前实现原则
- 只做**规则评分层**
- 不做复杂搜索树
- 不做 LLM 决策
- 不做正式联网同步

### 建议最小结构
- `DecisionContext`
- `ActionCandidate`
- `ScoreWeights`
- `AiPlanner`
- `AiProfiles`

---

## 3.2 第二优先：bot 队伍验证伪联机关系
目标：
让你单人就能测试第二支队伍，不依赖真人在线。

### 要验证什么
- 合作任务是否成立
- 竞争任务是否成立
- 合作后反转是否成立
- 撤离压力下 bot 是否会做合理选择

### 第一批 bot profile
- `coop_squad`
- `aggressive_rival`
- `reversal_squad`

### 第一批 bot 能力边界
应当会：
- 靠近任务目标
- 基础移动与攻击
- 低血保命
- 争关键点
- 在反转阶段切换行为

不应当会：
- 多回合精算
- 高级连携博弈
- 复杂 bait
- 高级心理战

---

## 3.3 第三优先：Boss AI
目标：
让 Boss 从“高血单位”变成任务核心锚点。

### Boss 的作用
- 迫使短期合作
- 创造阶段压力
- 驱动任务关系变化
- 与撤离节奏咬合

### 推荐第一批 Boss pattern
- `anchor_boss`
- `sweeper_boss`
- `summoner_boss`

### 当前阶段要求
- 仍然是**规则评分层 Boss**
- 追求可读、可压迫、可验证
- 不追求“像人一样聪明”

---

## 3.4 第四优先：野怪 / 中立单位层
目标：
让战场变成非纯 PVP，而是带第三方干扰的多边局。

### 推荐第一批 archetype
- `lurker`
- `guard`
- `hunter`
- `scavenger`

### 它们的作用
- 打断节奏
- 逼走位
- 增加不稳定性
- 让任务关系更复杂

### 为什么排在 Boss 后面
Boss 更直接服务核心任务验证；
野怪更像第二层调味，不该抢在 Boss 前面。

---

## 4. 下一阶段要补，但不是第一优先

## 4.1 编队 / 出战入口 UI
当前 BattleScene 仍有硬编码测试队伍逻辑，
后续应补成：

- 选底盘
- 装模块
- 校验 loadout
- 生成双方队伍
- 进入 run

这一步很重要，但它是为 bot 验证服务的，
不是先于 bot AI 的绝对第一优先。

## 4.2 信息遮罩 / recon visibility
这是伪联机感的重要来源。

目标：
- 开局只给公共信息
- 任务揭晓后再公开关键目标
- 队伍私有信息和公共信息分层

但这一步应在 bot 双队基础可跑后再加，
否则容易卡在 UI 细节里。

## 4.3 小规模 seed 地图 / 刷新规则
不是先做“AI 随机地牢”，
而是先做：

- 可复现 seed
- 固定模板上的变体
- 事件点刷新规则
- 小规模内容组合

目标是低成本模拟“每局略不同”，
而不是一开始就做重型生成系统。

---

## 5. 明确延后：未来增强项

## 5.1 LLM-enhanced Boss AI
这是**未来卖点候选**，但不进当前 MVP 后续第一阶段。

### 当前决定
- 现在不接 DeepSeek / GPT / Kimi 直接控制 Boss 回合
- 现在先把规则评分层 Boss 跑稳

### 未来推荐架构
- **规则层保底**：
  - 合法动作
  - 范围与冷却
  - 数值和护栏
  - 防犯蠢
- **LLM 层做高层意图**：
  - 阶段策略
  - 目标优先级
  - 战术人格
  - 台词风格

### 为什么延后
- 现在最重要的是玩法验证，不是模型接入
- 规则层不稳时接 LLM，调试会非常痛苦
- 需要先建立纯规则 Boss baseline，后面才好比较“LLM 是否真的更有趣”

### 未来要回答的问题
- LLM Boss 是否真的减少背板，而不是只增加随机性
- 是否会破坏公平性
- 调用频率、成本、时延是否可接受
- 是否值得成为核心卖点而不是实验模式

---

## 5.2 真联机 / 房间 / 服务端
明确延后。

原因：
- 当前最需要验证的是玩法关系，不是网络工程
- 真联机会显著抬高开发成本与调试复杂度
- 伪联机验证先跑通更划算

---

## 5.3 重型 AI 随机地牢生成
明确延后。

原因：
- 生成系统太重
- 当前没有必要先做内容生产大系统
- 在关系模型未验证前，先做大量生成内容风险高

正确顺序应是：
- 先 seed + 小变体
- 再半程序化
- 最后再考虑大模型/复杂生成接入

---

## 5.4 faux-2.5D / FFTA 风格表现升级
有价值，但不是当前最关键项。

### 为什么延后
- 它提升卖相，但不验证最核心玩法风险
- 当前更重要的是关系模型、Boss 压力、伪联机验证

### 什么时候值得提上来
当下面这些都成立时再上：
- bot 双队打起来成立
- Boss/野怪压场成立
- run 节奏稳定
- 需要更强 demo 卖相

---

## 6. 不要现在碰的坑

当前阶段不要优先投入：

- 正式匹配系统
- 权威服务器
- 防作弊
- 复杂行为树 / 搜索树 AI
- 全面程序化地图生成
- 纯 LLM 直接接管战斗动作
- 过早美术大重构

这些东西不是没用，
而是**现在做会拖慢真正关键的验证。**

---

## 7. 建议执行顺序

### 阶段 A：关系验证基础
1. 统一 AI foundation
2. bot 队伍 profile 接入
3. 单人 + bot 跑通三类 mission

### 阶段 B：PvE 压力成立
4. Boss AI 接入
5. Boss 驱动合作 / 反转 / 撤离验证
6. 野怪 / 中立单位接入

### 阶段 C：从验证走向更像产品
7. 编队入口 / 出战 UI
8. 信息遮罩 / recon visibility
9. seed 地图 / 事件刷新

### 阶段 D：增强项
10. faux-2.5D 表现升级
11. 半程序化内容扩展
12. LLM Boss 实验版
13. 真联机验证

---

## 8. 一句话版本

### 当前执行原则
**先做规则评分层的统一非玩家 AI，优先验证 bot 双队 + Boss + 野怪是否能把“合作 / 对抗 / 反转 / 撤离”这套核心循环真正撑起来。**

### 当前不做
- 不先上真联机
- 不先上 LLM Boss
- 不先上重型随机地牢
- 不先上大规模视觉重构

### 当前最优先
**统一 AI foundation → bot 队伍 → Boss → 野怪**
